## Режим экономии токенов

- Не писать промежуточные сообщения без причины. Сообщать только финал, реальный блокер или риск сломать сайт.
- Финальный отчет держать коротким: что изменено, какие проверки прошли, есть ли риски, можно ли принимать.
- Не вставлять длинные логи, HTML, diff, полные промты или большие списки файлов. Для проверок достаточно результата: прошло/не прошло и главная ошибка.
- Если нужен статус соседнего агента, читать только последний turn и без outputs, если нет особой причины.
- QA делает короткий smoke-check по согласованному чеклисту, без подробного протокола браузера.
- Manager сжимает отчеты рабочих агентов и отправляет lead только общий итог.
- Повторные/дублирующиеся промты считать одним и тем же заданием; не отвечать на дубликат отдельным отчетом.
- Перед началом работы читать `PROJECT_STATE.md`, если он есть, вместо перечитывания длинной истории чатов.
- Для нового этапа использовать короткий чеклист на 5-8 пунктов. Полные большие промты нужны только для новой роли или сложной архитектурной смены.
- Если задача требует подробностей, держать их в рабочем чате агента; lead получает только сжатый итог.
